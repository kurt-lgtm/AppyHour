<objective>
Implement the commit_pending flag and verification read that protect against the orderEditCommit error-but-applied pitfall. Before committing, the order ID is written to commit_pending in the checkpoint. If commit returns an error, a 2-second sleep followed by a verification read determines whether the edit actually applied — moving the order to succeeded or failed accordingly.

Purpose: Without this, a network error on orderEditCommit followed by a retry would add duplicate line items. This is Pitfall 1 from PITFALLS.md and the primary correctness risk for live sync (SYNC-05).

Output: _verify_edit_applied() helper, VERIFY_LINE_ITEMS_QUERY constant, and commit_pending checkpoint writes integrated into sync_order_to_shopify.
</objective>

    ```python
    def _verify_edit_applied(
        base: str, headers: dict, order_gid: str, expected_skus: list[str]
    ) -> bool:
        """Return True if all expected_skus appear on the order with fulfillableQuantity > 0.

        Used after orderEditCommit returns an error to detect the error-but-applied case (D-12).
        Reads current line items via GraphQL and checks whether the edit went through server-side.
        """
        data = _shopify_graphql(base, headers, VERIFY_LINE_ITEMS_QUERY, {"id": order_gid})
        order_data = data.get("order") or {}
        edges = (order_data.get("lineItems") or {}).get("edges") or []
        current_skus = {
            edge["node"]["sku"]
            for edge in edges
            if (edge.get("node") or {}).get("fulfillableQuantity", 0) > 0
        }
        return all(sku in current_skus for sku in expected_skus)
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Shopify GraphQL verification read response → _verify_edit_applied | Response from live API; could be malformed or empty |
| commit_pending list in checkpoint → retry skip logic | Read from disk; could contain stale IDs from a previous pipeline_id |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Tampering | _verify_edit_applied response | mitigate | Use .get() with safe defaults on all nested dict access; missing fields return False (conservative — marks failed rather than succeeded) |
| T-02-08 | Tampering | stale commit_pending across pipeline_ids | accept | pipeline_id in PipelineState ties checkpoint to a specific run; operator starting a new run generates new pipeline_id, clearing stale state |
| T-02-09 | Information Disclosure | error messages in errors dict | accept | errors dict stored in local checkpoint file only; not transmitted; contains Shopify API error strings which contain no user PII |
| T-02-10 | Denial of Service | 2s sleep per error-but-applied order | accept | Only triggered on commit errors (uncommon); maximum delay = error_count * 2s; acceptable for a CLI tool |
</threat_model>
